## 2. Core Rules

These rules apply to every day of the program, with no exceptions:

1. **Fixed 9 daily modules**, always in this order:
   1. Review previous days
   2. New vocabulary
   3. Grammar / sentence pattern
   4. New sentences
   5. Speaking practice
   6. Listening / shadowing
   7. Java technical English
   8. Java senior interview
   9. Daily summary
2. **Playable pronunciation (mandatory):** every single English word,
   sentence, question, and answer used in this program must be tagged
   `🔊 [PLAY]` immediately before it. When working with ChatGPT (or any
   assistant), every tagged item must actually be played as audio/voice
   before moving on — nothing tagged `🔊 [PLAY]` should be silently skipped.
3. **Java interview answer framework (mandatory):** every Java interview
   answer must follow this exact 8-step structure, in order, every time,
   with no steps skipped:
   1. **Analysis**
   2. **Eliminate**
   3. **Suspect**
   4. **Mitigation**
   5. **Root Cause**
   6. **Optimization**
   7. **Prevention**
   8. **Lessons Learned**

   Short mnemonic: **A-E-S-M-R-O-P-L**.
4. **Review before new content:** each day must start with Module 1 (review
   of previous days) before any new vocabulary, grammar, or Java content is
   introduced.
5. **Update this file at the end of every day:** Section 3 (Current Session
   Memory), Section 4 (Master Progress Tracker), and the relevant banks
   (Sections 5–9) must be updated before the day is considered closed. See
   Section 11 (Update Protocol).
6. **Single file, no new files:** `ENGLISH_JAVA_90DAYS_SOT.md` is the only
   learning-record file for this project. No new learning-record md files
   may ever be created.
7. **SOT Consistency Check before ending any chat:** before ending any chat,
   this file must first be updated with everything learned in the session,
   then a Consistency Check must be run. If any learning content is not yet
   synced into this file, it must be updated first — never end the chat and
   fix it later. Output:
   `SOT Consistency Check`
   `PASS` or `FAIL`

## 9. Java Interview Bank

Covers Module 7 (Java technical English) and Module 8 (Java senior
interview) of the daily structure.

### 9.1 Mandatory Answer Framework

Every Java interview answer must use this exact structure:

1. **Analysis**
2. **Eliminate**
3. **Suspect**
4. **Mitigation**
5. **Root Cause**
6. **Optimization**
7. **Prevention**
8. **Lessons Learned**

Answer template:

```
🔊 [PLAY] Question: <interview question>

1. Analysis: ...
2. Eliminate: ...
3. Suspect: ...
4. Mitigation: ...
5. Root Cause: ...
6. Optimization: ...
7. Prevention: ...
8. Lessons Learned: ...
```

Every line of the answer (question + all 8 steps) is an English item and
must be tagged `🔊 [PLAY]` when actually drilled in a session.

### 9.2 Worked Example (framework demonstration only)

🔊 [PLAY] Question: "The application suddenly becomes slow under high load. How do you approach this?"

1. **Analysis** — 🔊 [PLAY] "First, I check when the slowdown started and what changed: traffic, deployment, or data volume."
2. **Eliminate** — 🔊 [PLAY] "I rule out network issues and client-side problems by checking server-side metrics first."
3. **Suspect** — 🔊 [PLAY] "I suspect a database connection pool bottleneck or a slow query introduced recently."
4. **Mitigation** — 🔊 [PLAY] "I temporarily increase the connection pool size and add a query timeout to stabilize the system."
5. **Root Cause** — 🔊 [PLAY] "The root cause is a missing index on a frequently queried column, causing full table scans."
6. **Optimization** — 🔊 [PLAY] "I add the missing index and introduce caching for the hot query."
7. **Prevention** — 🔊 [PLAY] "I add query performance monitoring and alerts for slow queries in the future."
8. **Lessons Learned** — 🔊 [PLAY] "Always monitor query performance as data grows, not just at launch."

**Status:** This is a framework demonstration, not tied to a specific day. It
shows the required structure so any real interview question logged under a
specific day (below) can follow the same shape.
